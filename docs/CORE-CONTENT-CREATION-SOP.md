## Document Purpose and Scope

This document establishes the foundational standards, methodology, and requirements for all content creation at ASP Branding. Every team member and AI tool must follow these standards without exception.

This Core SOP works in conjunction with:
- Vertical Playbooks (Home Services, Legal Services)
- Service-Specific Modules (HVAC, Plumbing, Home Inspection, Family Law, Personal Injury, etc.)
- Individual Client Project Guidelines

**Document Hierarchy (highest to lowest priority):**
1. Client-specific project guidelines
2. Service-specific modules
3. Vertical playbooks
4. This Core SOP

When conflicts arise, follow the higher-priority document. Client-specific guidelines may add requirements but cannot override the verification and accuracy standards contained in this Core SOP.

## Section 1: Absolute Content Rules

These rules apply to ALL content creation without exception. Violation of these rules requires immediate correction before any content is published or delivered.

### 1.1 Information Verification Standard

#### Rule 1: No Unverified Information

All factual claims, statistics, company information, service details, pricing, credentials, and business data must be verified against:
- Client-provided documentation in the project files
- Official client website (verified URLs only)
- Authoritative third-party sources (government sites, industry associations, established publications)

**This means NEVER:**
- Invent company history, founding dates, or credentials
- Fabricate customer testimonials or case studies
- Create fictional statistics or data points
- Assume service offerings, pricing, or service areas
- Guess at certifications, licenses, or awards
- Create fictional customer stories or scenarios

#### Rule 2: When Information Is Missing

If content cannot be created accurately due to missing information, the content creator (human or AI) must:

1. Stop content creation at that point
2. Clearly identify what specific information is needed
3. Request the information using the standard format below
4. Wait for verified information before proceeding

**Required Information Request Format:**

```
INFORMATION NEEDED TO CONTINUE

Content Piece: [Title/Description]
Section: [Where in the content]

Missing Information:
- [Specific item 1]
- [Specific item 2]

Why This Is Needed:
[Brief explanation of why this cannot be assumed or omitted]

Suggested Sources:
- [Where this information might be found]
```

#### Rule 3: No Filler Content

Content must never include:
- Generic statements that add no value
- Placeholder text intended to be replaced later
- Vague claims without specifics
- Padding to reach word counts
- Repetitive information restated for length

If word count targets cannot be met with verified, valuable information, reduce the target rather than add filler.

---

### 1.2 Project Guidelines Compliance

#### Rule 4: Project Files Are Mandatory Reference

When working within a Claude Project or any structured content environment:

1. All files added to the project are considered authoritative sources
2. Content must align with information in project files
3. Project-specific guidelines override general practices
4. URLs provided in project files are the only verified URLs for that client

**Before Creating Any Content:**
- Review all project files for relevant information
- Identify client-specific requirements or restrictions
- Note any brand voice, terminology, or style guidelines
- Confirm service offerings, service areas, and business details
- Identify customer journey stage the content serves

#### Rule 5: URL Verification Standard

**For Client Websites:**
- Only use URLs that exist in project files or have been explicitly provided
- Never construct URLs by guessing page structures
- Never assume a page exists because it logically should
- If a needed page URL is not available, request it

**For External Links:**
- Only link to verified, accessible URLs
- Confirm the linked page contains relevant, current information
- Never link to paywalled content without notation
- Never link to pages that may have moved or been removed

**URL Verification Checklist:**
- [ ] URL exists in project files OR has been explicitly provided
- [ ] URL is correctly formatted
- [ ] For external links: Source is authoritative and current
- [ ] Link context accurately describes destination content

## Section 2: Content Structure Standards

### 2.1 Intent-First Architecture

All content must be structured around user intent, following the decomposition principles that modern AI systems use to understand content relevance.

#### The Three-Layer Model

**Layer 1: Atomic (Section Level)**

Each H2 section must be:
- Self-contained and understandable independently
- Focused on one specific user need or question
- Summarizable in one to two sentences
- Actionable or informative on its own

**Layer 2: Aggregate (Page Level)**

The collection of sections must:
- Form a coherent journey toward the page's purpose
- Progress logically from one section to the next
- Cover the complete user need without gaps
- Lead to a clear outcome or next step

**Layer 3: System (Site Level)**

Content pieces must:
- Connect to related content through strategic internal linking
- Serve distinct purposes without excessive overlap
- Guide users through complete decision journeys
- Support both entry at any point and linear progression

#### Section Structure Template

Every H2 section should follow this internal pattern:

**Context (1-2 sentences):** Why this information matters or what situation prompts this question

**Core Content (main body):** The actual information, explanation, or guidance

**Action Bridge (1-2 sentences):** What the reader should do with this information or what they will naturally want to know next

#### Content Validation Test

Before finalizing any page, apply this test:
1. Summarize each H2 section in one sentence
2. Combine all summaries into a single paragraph
3. Verify the paragraph accurately describes the page's complete purpose
4. If it doesn't, restructure until it does

---

### 2.2 Customer Journey Alignment

All content must be tagged and structured according to its position in the customer journey.

#### Journey Stages

**Awareness Stage**
- User recognizes they have a problem or need
- Content focus: Education, problem identification, symptom recognition
- Intent: Understanding what's happening and why it matters

**Consideration Stage**
- User researches solutions and options
- Content focus: Comparisons, process explanations, option evaluation
- Intent: Understanding available solutions and their tradeoffs

**Decision Stage**
- User ready to select a provider or take action
- Content focus: Trust signals, differentiators, conversion facilitation
- Intent: Choosing the right solution and taking action

#### Journey Stage Requirements

Every content piece must:
- Be explicitly tagged with its primary journey stage
- Contain appropriate content depth for that stage
- Link to content serving adjacent journey stages
- Include conversion elements appropriate to the stage

### 2.3 Linking Requirements

#### Internal Linking Standard

**Minimum internal links per content piece:**
- Blog posts: 4 internal links
- Service pages: 6 internal links
- Hub pages: 8 internal links (plus links to all associated spokes)
- Landing pages: 2 internal links

**Internal Link Placement Rules:**
- Links must appear at natural intent transition points
- Anchor text must clearly signal what the linked page provides
- Links must lead to content that fulfills the signaled intent
- Distribute links throughout content, not clustered in one section
- First internal link should appear within the first 300 words

**Anchor Text Standards:**

Good: `Compare the costs of AC repair versus full system replacement`
(Specific intent signal, user knows what they will find)

Bad: `Click here for more information`
(No intent signal, generic text)

Bad: `Learn more about our services`
(Vague, doesn't help user or search engines)

#### External Linking Standard

**Minimum external links per content piece:**
- Blog posts: 2 external links
- Service pages: 1 external link
- Hub pages: 3 external links

**Approved External Link Sources:**

*Tier 1 - Preferred (use these first):*
- Government websites (.gov domains)
- Educational institutions (.edu domains)
- Wikipedia (for general reference and definitions)
- Official industry associations
- Peer-reviewed publications

*Tier 2 - Acceptable:*
- Major news organizations (NYT, WSJ, Reuters, AP)
- Established industry publications
- Manufacturer official websites
- Professional licensing boards

*Tier 3 - Use Sparingly:*
- Major business publications (Forbes, Inc.)
- Established trade publications
- Reputable research firms

**Never Link To:**
- Competitor websites
- Low-quality or thin content sites
- Sites with excessive advertising
- Paywalled content (without notation)
- Social media posts (except for specific attribution)
- Content farms or article directories

### 2.4 AI Discoverability Standards

To optimize content for AI systems and answer engines, include structured elements that facilitate content understanding and extraction.

#### Key Takeaway Blocks

Include "Key Takeaway:" summary blocks in all long-form content:
- After each major section (H2) in hub pages
- After every 2-3 sections in blog posts
- At minimum, one per 500 words of content

**Format:**
```
**Key Takeaway:** [One to two sentence summary of the essential 
point from the preceding section. Must be understandable without 
reading the full section and provide genuine value on its own.]
```

**Key Takeaway Requirements:**
- Must accurately summarize the preceding content
- Must be understandable without context
- Must provide actionable or valuable information
- Must not be generic or obvious statements
- Must not simply repeat the section heading

**Good Example:**
```
**Key Takeaway:** Most residential AC systems last 15-20 years, 
but units over 10 years old needing repairs costing more than 
50% of replacement value should typically be replaced rather 
than repaired.
```

**Bad Example:**
```
**Key Takeaway:** It's important to maintain your AC system regularly.
```

#### FAQ Sections

All blog posts and service pages must include an FAQ section.

**Minimum FAQ questions:**
- Blog posts: 6 questions
- Service pages: 8 questions
- Hub pages: 10 questions

**FAQ Requirements:**
- Questions must serve the page's primary intent
- Questions must be ones real users would ask
- Answers must be comprehensive but concise (50-150 words typical)
- Questions should follow logical progression matching customer journey
- Schema markup required for all FAQ sections
- All answers must contain only verified information

**Question Selection Checklist:**
- [ ] Would someone with this page's intent ask this question?
- [ ] Does the answer advance the user toward the page goal?
- [ ] Is this question specific enough to be useful?
- [ ] Does this question avoid overlap with other FAQs on this page?
- [ ] Can this question be answered with verified information only?

#### Comparison Blocks

For content in the consideration stage, include structured comparison elements:
- Side-by-side option comparisons where relevant
- Clear criteria for evaluation
- Objective presentation of tradeoffs
- Links to detailed information on each option

## Section 3: Content Quality Standards

### 3.1 Writing Standards

#### Voice and Tone
- Follow client-specific voice guidelines when provided
- Default to conversational but professional tone
- Write for the target audience's knowledge level
- Avoid jargon without explanation
- Use active voice predominantly

#### Readability
- Target 8th-10th grade reading level for consumer content
- Vary sentence length (mix short, medium, and occasional longer sentences)
- Keep paragraphs to 3-4 sentences maximum
- Use subheadings every 200-300 words
- Include transitional phrases between sections

#### Formatting Restrictions
- Maximum 2 bulleted or numbered lists per content piece
- Lists should contain 4-7 items
- Use bold sparingly for genuine emphasis only
- Avoid walls of text without visual breaks
- Include relevant images or visual elements where appropriate

#### Word Count Guidelines

| Content Type | Target Range | Minimum | Maximum |
|--------------|--------------|---------|---------|
| Blog Post | 1,500-2,200 | 1,200 | 2,500 |
| Service Page | 800-1,500 | 600 | 2,000 |
| Hub Page | 3,000-5,000 | 2,500 | 6,000 |
| Landing Page | 500-1,000 | 400 | 1,500 |
| Location Page | 600-1,000 | 500 | 1,200 |

**Important:** Word counts are targets, not mandates. Never add filler to reach word counts. Reduce targets if verified content cannot support them.

---

### 3.2 SEO Standards

#### On-Page Elements

Every content piece requires:
- Title tag (under 60 characters, includes primary keyword)
- Meta description (under 160 characters, compelling and accurate)
- H1 heading (matches or closely aligns with title tag)
- Logical H2/H3 hierarchy (no skipping heading levels)
- Primary keyword in first 100 words
- URL slug (concise, includes primary keyword, lowercase with hyphens)
- Publication date and last modified date

#### Keyword Integration
- Primary keyword density: 1-2% (natural integration only)
- Include keyword variations and semantically related terms
- Use keywords in at least one H2 heading
- Include keywords in image alt text where relevant
- Never force keywords unnaturally
- Include NLP-relevant related terms throughout content

#### Image Requirements
- All images must have descriptive alt text
- Alt text should include relevant keywords naturally
- File names should be descriptive (not IMG_1234.jpg)
- Images should be compressed for web performance
- Include at least one relevant image per 500 words of content

#### Schema Markup

**Required schema types by content:**

| Content Type | Required Schema |
|--------------|-----------------|
| Blog posts | Article, FAQPage, BreadcrumbList |
| Service pages | Service, LocalBusiness, FAQPage |
| Hub pages | Article, FAQPage, BreadcrumbList, ItemList |
| Location pages | LocalBusiness, Service, FAQPage |

All schema must be validated using Google's Rich Results Test before implementation.

---

### 3.3 Local SEO Standards

#### Geographic References

For location-based businesses:
- Mention primary service city in first 100 words
- Include service area naturally throughout content
- Reference local landmarks, neighborhoods, or conditions where relevant
- Include city/region in at least one H2 heading
- Use location-modified keywords appropriately

#### NAP Consistency

Business name, address, and phone must:
- Match exactly across all content
- Use the format established in project files
- Never abbreviate inconsistently
- Use consistent phone number formatting

#### Local Content Authenticity

Content must:
- Reference genuine local conditions (climate, building types, regulations)
- Mention relevant local codes or requirements when applicable
- Include local expertise signals where verifiable
- Avoid generic content that could apply anywhere

## Section 4: Entity and Authority Standards

### 4.1 Entity Optimization

All content must establish clear entity relationships for AI systems and knowledge graphs.

#### Entity Introduction Standard

Within the first paragraph of any content, include:
- Full entity name (business name or author name)
- Entity type indicator (business description, professional title)
- Geographic association (location, service area)
- Authority signal (experience, credentials, recognition)

**Example:**
```
Great Lakes Heating & Cooling, a licensed HVAC contractor serving 
Metro Detroit since 2012, provides comprehensive residential and 
commercial heating and cooling services including installation, 
repair, and maintenance throughout Wayne, Oakland, and Macomb counties.
```

#### Entity Reinforcement Requirements

- Reference the primary entity by name minimum once per 300 words
- Use consistent naming throughout (don't randomly switch between "the company" and brand name)
- Include entity in schema markup
- Connect entity to sameAs profiles

#### sameAs Platform Integration

Every content piece must include at least 2 references to the entity's presence on other platforms:
- Website references where appropriate
- Social media profile mentions where natural
- Directory listing references where relevant
- Industry association memberships

These references can be direct (linking to profiles) or contextual ("Connect with us on Facebook" or "See our reviews on Google").

---

### 4.2 E-E-A-T Demonstration

All content must demonstrate Experience, Expertise, Authoritativeness, and Trustworthiness through verifiable signals only.

#### Experience Signals
- Reference actual work performed (verified only)
- Include perspective that comes from doing the work
- Mention timeframes and volume where verifiable
- Share practical insights from hands-on experience

#### Expertise Signals
- Cite credentials and certifications (verified only)
- Demonstrate technical knowledge appropriate to topic
- Explain concepts accurately and thoroughly
- Reference training or specialized knowledge

#### Authoritativeness Signals
- Link to authoritative external sources
- Reference industry standards and best practices
- Include relevant data and statistics with sources
- Position content within broader industry context

#### Trustworthiness Signals
- Maintain transparency about limitations
- Present balanced information (pros and cons)
- Include clear contact information and business details
- Avoid exaggerated claims or promises

#### What We Never Fabricate
- Years of experience
- Number of projects completed
- Customer testimonials or reviews
- Awards or recognition
- Certifications or licenses
- Case study details
- Statistical claims
- Customer stories or scenarios

## Section 5: Content Types and Specifications

### 5.1 Blog Post Specifications

#### Structure

```
TITLE (H1)
├── Include primary keyword
├── Compelling and specific
└── Under 60 characters preferred

INTRODUCTION (100-150 words)
├── Hook: Address the reader's situation or question
├── Context: Brief background on why this matters
├── Promise: What the reader will learn or be able to do
└── Entity introduction: Establish who is providing this information

BODY SECTIONS (H2s, 4-6 sections)
├── Each section: 200-400 words
├── Each section follows Context-Content-Bridge structure
├── Key Takeaway after major sections
└── Internal links distributed throughout

FAQ SECTION
├── Minimum 6 questions
├── Questions serve the post's primary intent
├── Comprehensive but concise answers
└── FAQPage schema applied

CONCLUSION (100-200 words)
├── Summarize key points
├── Clear call to action
├── Next steps for the reader
└── Contact or conversion opportunity
```

#### Blog Post Checklist

```
Information Verification:
[ ] All claims verified against project files
[ ] No fabricated statistics or data
[ ] No fictional customer stories
[ ] All company information accurate

Structure:
[ ] Title includes primary keyword
[ ] Primary keyword in first 100 words
[ ] Entity introduced in first paragraph
[ ] 4-6 body sections with H2 headings
[ ] Context-Content-Bridge pattern in each section
[ ] Key Takeaway blocks included
[ ] 6+ FAQ questions

Linking:
[ ] 4+ internal links distributed throughout
[ ] 2+ external links to Tier 1 sources
[ ] All URLs verified
[ ] Anchor text is descriptive

SEO:
[ ] Meta description under 160 characters
[ ] URL slug is clean and keyword-focused
[ ] Images have alt text
[ ] Schema markup prepared

Quality:
[ ] Maximum 2 bulleted lists
[ ] Word count 1,500-2,200 (no filler)
[ ] Clear call to action in conclusion
[ ] Brand voice consistent
```

### 6.2 Content Review Workflow

#### Stage 1: Draft Creation
- Content created following all SOP standards
- All required elements included
- Information gaps identified and flagged
- Content paused if missing critical information

#### Stage 2: Information Verification
- All claims checked against project files
- Missing information requested and obtained
- URLs verified as working
- Statistics confirmed with sources

#### Stage 3: Quality Review
- Readability and flow assessed
- Brand voice consistency checked
- SEO elements verified
- Linking requirements confirmed

#### Stage 4: Technical Implementation
- Schema markup added and validated
- Proper formatting applied in CMS
- Images optimized and alt text added
- Internal links connected

#### Stage 5: Final Approval
- Complete checklist reviewed
- Stakeholder approval obtained
- Publication scheduled

Content cannot advance to the next stage if information gaps remain or verification is incomplete.

## Section 7: AI Tool Usage Standards

### 7.1 Working with Claude and AI Tools

#### Project Context Requirements

Before using AI for content creation:
- Ensure all relevant project files are loaded
- Provide complete business information
- Include brand voice guidelines
- Specify all requirements upfront

#### Required Prompt Elements

When requesting content from AI tools, always include:

```
1. Project guideline reference:
   "Follow all project files and guidelines for [Client Name]"

2. SOP compliance:
   "Follow all standards in the Core Content Creation SOP"

3. Content specifications:
   "Create a [content type] of [word count] words about [topic]"

4. Structural requirements:
   "Include [number] internal links, [number] external links to 
   Tier 1 sources, Key Takeaway blocks, and [number] FAQ questions"

5. Verification instruction:
   "Only use verified information from project files. If any 
   required information is missing, stop and ask for it rather 
   than assuming or fabricating. Never create fictional customer 
   stories, statistics, or credentials."

6. Journey stage:
   "This content serves users in the [awareness/consideration/decision] 
   stage of their journey."
```

#### AI Output Requirements

All AI-generated content must:
- Be reviewed for accuracy before use
- Have all factual claims verified against project files
- Have all suggested links checked for validity
- Be confirmed for brand voice consistency
- Never be published without human review

### 7.2 Information Request Protocol

When AI tools or team members identify missing information:

#### Triggers for Information Request
- Company details not in project files
- Statistics or data without sources
- Service offerings that cannot be verified
- Geographic information needing confirmation
- Credentials or certifications not documented
- Pricing or timeline information not provided
- URLs that cannot be verified

#### Standard Request Format

```
CONTENT CREATION PAUSED - INFORMATION NEEDED

Project: [Client Name]
Content Piece: [Title/Description]
Created By: [Name/AI Tool]
Date: [Date]

MISSING INFORMATION:

1. [Specific item needed]
   - Why needed: [Explanation]
   - Where it appears: [Section]
   - Suggested source: [Where to find it]

2. [Specific item needed]
   - Why needed: [Explanation]
   - Where it appears: [Section]
   - Suggested source: [Where to find it]

CONTENT STATUS:
Creation paused. Sections requiring this information marked 
with [NEEDS VERIFICATION] in draft.

DEADLINE IMPACT:
[Note if this affects delivery timeline]
```

#### Response Requirements
- Information requests addressed within 24-48 hours
- Responses include source documentation
- If information unavailable, content scope adjusted
- Never proceed with assumed information

- Wall Street Journal
- Reuters
- Associated Press

### Never Link To
- Competitor websites
- Content farms
- Sites with excessive ads
- Social media posts (except attribution)
- Forums or user-generated sites
- Any site with questionable authority
